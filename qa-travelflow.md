## PRIORITY BUG: Destination Edits Not Sticking

### Root Cause (confirmed via code analysis)

**The `loadTripsFromJson()` function races against and overwrites user edits.**

Here's what happens:

1. User opens the app → `loadTrips()` runs (line ~2128)
2. It immediately calls `loadTripsFromJson()` **asynchronously** (no await, fire-and-forget)
3. The user edits a destination and saves → `saveTrips()` writes to localStorage AND calls `saveToGitHub()`
4. Meanwhile `loadTripsFromJson()` finishes fetching `trips.json` from GitHub and calls `this.saveTrips()` and `this.renderAllViews()` (line ~2213-2215)
5. **The `loadTripsFromJson` merge logic re-applies the `travelflow-user-edits` store**, but there's a critical gap:

**The real failure mode:**
The `saveToGitHub()` function (line ~3558-3585) **strips many fields when writing back to GitHub**. It only saves:
```
id, title, destination (restructured), dates, status, type, bookings, tasks, companions, notes
```
It **does NOT save**: `flights` (in the new format), `accommodation`, `budget`, `expenses`, `packingList`, `colorIndex`

So when the page next loads and `loadTripsFromJson()` fetches `trips.json`:
- The JSON on GitHub is missing fields that were saved only in localStorage
- The merge uses `userEdits` store — but `saveToGitHub` being called from `saveTrips()` means the GitHub copy is always behind
- **The `destination` field is reconstructed** from `dest.city + ', ' + dest.country` (line ~2140), so if you saved "New York, USA" it becomes "New York, USA" — but if the GitHub file has it as `{city: "New York", country: "USA"}` and your edit was "New York City, USA", the round-trip strips your edit

**Secondary cause: `initDashboard()` is called twice on startup** (lines 2120-2121):
```js
this.renderAllViews();
this.initDashboard();
this.initDashboard(); // ← duplicate call
```
And in `loadTripsFromJson()` after save:
```js
this.saveTrips();
this.renderAllViews();
this.initDashboard(); // ← third call
```
This can cause partially-updated state to be rendered before async operations complete.

**Tertiary cause: `packingList` type mismatch**
- When trips come from `trips.json`, `packingList` defaults to `[]` (array, line 2176)
- But the edit form saves `packingList` as an object `{}` (line 2731+)
- Code later checks `Object.keys(trip.packingList).length > 0` (line 3092) — this throws if `packingList` is an array
- Some code paths treat it as array (`trip.packingList.filter(...)` lines 3671-3672), others as object
- **This mismatch can cause silent failures that prevent the full trip save from completing**

### Step-by-Step Reproduction
1. Load TravelFlo → trips load from JSON
2. Click edit on a trip → change the destination name
3. Save the trip
4. Immediately (within ~2 seconds), or on next page load, destination reverts
5. **Worst case**: if `loadTripsFromJson()` async fetch completes ~500ms after your save, it overwrites your edit with the GitHub version before `saveToGitHub()` has had a chance to commit

### Fix

**Short-term (localStorage-only fix):**
```js
// In loadTripsFromJson(), before overwriting this.trips:
// Check if a save is in-progress; if so, skip the merge
if (this._saving) return;
```

**Proper fix:**
1. Make `saveToGitHub()` write ALL fields (not a reduced subset)
2. Add a `_jsonLoading` flag; skip async merge if user made edits since load started
3. Remove duplicate `initDashboard()` calls (lines 2120-2121)
4. Normalize `packingList` to always be an object `{}` on load

## OTHER BUGS

### BUG-T1: `saveToGitHub()` strips critical fields on write
**Severity: Critical**
**Lines: 3568-3584**

The PUT to GitHub only includes: `id, title, destination, dates, status, type, bookings, tasks, companions, notes`. Missing: `flights`, `accommodation`, `budget`, `expenses`, `packingList`.

Every save to GitHub loses these fields. Next load reconstructs them from `trips.json` (which is now missing them), causing data loss.

**Fix:** Include all trip fields in the serialized output, OR maintain a separate "GitHub format" for `trips.json` that stores the full app state.

### BUG-T3: `packingList` type inconsistency causes crashes
**Severity: High**
**Lines: 2176, 2739, 3092, 3671-3672, 4008, 4022, 4143**

Some code paths expect `packingList` to be an `Array`, others expect an `Object`. Code at lines 4008 and 4022 calls `trip.packingList.filter(...)` — this fails if `packingList` is `{}`. The render at line 3092 calls `Object.keys(trip.packingList).length > 0` — this fails if it's `[]`.

**Fix:** Normalize `packingList` to a consistent type (recommend `{}` keyed by category) on every load path.

### BUG-T6: Expense category parsing is fragile
**Severity: Medium**
**Lines: 2718-2722**

When re-reading expense items from the DOM to save:
```js
const category = item.querySelector('.expense-category').textContent.toLowerCase();
expenses.push({ category: category.split(' ')[1] || 'other' });
```
This reads the DOM text (which includes an emoji + space + category name) and tries to extract category via `split(' ')[1]`. This breaks for multi-word categories or if the emoji rendering changes.

**Fix:** Store category as a `data-category` attribute on the element.

### BUG-T7: Token exposed in URL, no input field for manual entry
**Severity: High (security/UX)**
**Lines: 3558-3561**

The GitHub token is read from `?token=` URL param only. No fallback input, no masked storage. The token appears in browser history, server logs, and referrer headers.

## UX ISSUES

### UX-T1: No save confirmation / feedback
After editing a trip and saving, there's no visual confirmation that the save succeeded (either to localStorage or GitHub). Users don't know if their edit stuck.
**Fix:** Show a toast/notification "Trip saved ✓" and a subtle GitHub sync indicator.

### UX-T2: Edit modal doesn't show which trip you're editing
The modal title just says "✏️ Edit Trip" — no destination name. For users with many trips, it's disorienting.
**Fix:** Set title to "✏️ Edit Trip – [Destination]".

### UX-T3: No loading state during GitHub sync
The app fetches `trips.json` on every load but shows no loading indicator. If the fetch is slow, the UI flickers as views re-render after data arrives.
**Fix:** Show a subtle "Syncing..." state and only render after data is ready.

### UX-T4: Flight form fields don't match stored data keys
The form saves flights with `depAirport`, `arrAirport`, `depTime`, `arrTime` — but the existing data uses `from`, `to`, `departure`, `arrival`, `confirmation`. When editing an existing trip's flights, the fields will be empty because the keys don't match.
**Fix:** Normalize field names or map them in `populateTripForm()`.

### UX-T5: "Add New Trip" tab doesn't behave like a tab
Clicking the "Add" nav item opens a modal instead of navigating to a tab (line 2385-2388). This is inconsistent with the other nav items and breaks tab state.

### UX-T6: Discover tab renders 4+ times per navigation
Switching to discover tab calls `renderDiscover()`, `renderEnhancedDiscover()` twice, and `renderAllTrips()`. Page visibly flashes.

## SUGGESTIONS

1. **Add a "Saved ✓" indicator** — a small green checkmark near the header that lights up after each GitHub sync, with a timestamp of last save
2. **Show sync status** — "Last synced 2 min ago" in the header
3. **Conflict resolution UI** — if GitHub save fails due to conflict (409), show the user what changed and let them choose
4. **Trip card photos** — already partially implemented with Unsplash; extend to more destinations
5. **Keyboard shortcuts** — Esc to close modals, Cmd+S to save
6. **Mobile nav** — the bottom navigation is desktop-first; on mobile, consider a sticky bottom nav bar
7. **Budget tracking** — the budget progress bar is already there; add a "Add Expense" quick button directly on the trip card
8. **Dark/light mode toggle** — the app is dark-mode only; some users may prefer light
